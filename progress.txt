## Codebase Patterns
- Generated Go is committed under internal/**/*.go and gated by `task check` →
  `verify-generated`: it re-emits to a scratch tree and diffs against the tree.
  Any change to emit output requires re-running `task generate` so the committed
  files match, or the drift gate fails.
- `cmd/goal/main.go` is hand-written Go (NOT goal-generated). Edit it directly,
  then `task build` rebuilds bin/goal before `task generate` regenerates internal.
- The emit write path is `emitFiles` in cmd/goal/main.go; corpus/by-example tests
  compare in-memory pipeline output (gf.Go), not on-disk emitted files, so a
  write-time change (e.g. a header) does not affect those goldens.
- The project verify command is `task check` (vet + build + verify-generated +
  go test ./...). Zero-dependency: stdlib `testing` only, no testify.
- morse-code `history/` is gitignored — initiative artifacts are not committed.
- Emit-lifecycle ownership is keyed on the goal line-1 marker, defined ONCE as
  `goalGeneratedMarker` in cmd/goal/main.go; reuse `IsGoalGenerated` rather than
  re-deriving the marker. Refactoring an emitted literal into a same-valued
  constant keeps `verify-generated` green with zero regen (no internal/ churn).
- Adding/removing a subcommand or flag in `guideCommands` (cmd/goal/main.go)
  changes the AI guide output, so `TestBootstrapGoldenMatches` fails until you
  regenerate: `go run ./cmd/goal ai > AI-KNOWLEDGE-BOOTSTRAP.md`.
- Every top-level `testdata/*.goal` transpile pair is auto-COMPILED+VETTED by
  the corpus behavioral tier (internal/corpus/behavior_runner.go RunCompile,
  driven by TestCompileRunner): the generated Go is written into an isolated
  temp module and `go build`+`go vet` run against it. So a testdata pair must be
  a complete, self-contained, vet-clean program (all callees/types defined
  in-file). This gives free "the emitted Go compiles" coverage for new goldens;
  for actual runtime-value assertions, build+`go run` a `package main` fixture
  in a backendtest (see internal/backendtest/native_trailing_error_test.go).
- To add a testdata transpile pair: write `testdata/<n>.goal`, seed an EMPTY
  `testdata/<n>.go.expected`, `go run ./cmd/corpus-gen` (pairs it into the
  manifest), then `go test ./internal/corpus -run TestUpdateGoldens
  -update-goldens` to fill the golden, and bump the hardcoded transpile count in
  internal/corpus/generate_test.go (TestGenerateCounts).
