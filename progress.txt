# Progress Log — goal AST front-end rewrite

## Codebase Patterns
- Zero-dependency: stdlib only. Tests use stdlib `testing` (NO testify), even
  though the injected go.md rule mentions testify — the project constraint wins.
- New packages live under `internal/`; fixtures go in a sibling `testdata/` dir
  (see internal/{analyze,check,pipeline,corpus}/testdata).
- Verify gates (prd.json verifyCommands): `go build ./...`, `go vet ./...`,
  `go test ./... -count=1`. All must be green before committing.
- Follow REWRITE-ARCHITECTURE.md phasing; §10 details Phases 0–2 (the approved
  scope). Phase 0 = corpus hardening (US-001..US-010).
- Each story runs through the morse-code `feature` workflow in --automode on the
  current branch (no new branch); human-review gates are passed via
  `morse-code workflow resume`.
- The golden corpus is indexed by `corpus/manifest.json` (repo root). Regenerate
  after adding/removing golden files: `go generate ./internal/corpus` (or
  `go run ./cmd/corpus-gen -root .`). Paths are repo-root-relative, slash-form.
- `internal/corpus` imports `internal/pipeline` and `internal/check`. Any test
  that delegates to a corpus runner must therefore live in an EXTERNAL `_test`
  package (`package pipeline_test` / `check_test`) to avoid an import cycle; an
  internal-package test importing corpus will not compile. Internal and external
  test packages coexist fine in one directory. From `internal/<pkg>`, repo root is
  `../..` and the manifest is `../../corpus/manifest.json` (same depth as corpus).

## [2026-06-27T02:13:00Z] - US-003
- Implemented the interface-based transpile runner (Phase 0.3). Added a
  Transpiler interface{ Transpile(src string)(pipeline.Output,error) } plus a
  TranspilerFunc adapter so the free pipeline.Transpile satisfies it without
  changing pipeline. RunTranspile(root, Case, Transpiler) reads each manifest
  case's input + golden, transpiles through the interface, and compares after
  gofmt-normalizing BOTH sides (go/format.Source).
- A case passes when the normalized golden equals Output.Go OR (when non-empty)
  Output.Test. The sidecar fallback is required because the manifest classifies
  the feature-11 doctest examples as transpile cases whose golden is the emitted
  _test.go, while testdata/doctest_funcs (also has /// doctests) goldens the main
  Go output. Presence of doctests in the source does NOT decide which output the
  golden represents — only content does — so the runner accepts either.
- TestTranspileRunner drives all 51 transpile cases through pipeline.Transpile;
  all pass. It fails loudly (t.Fatalf) if the manifest yields zero cases.
- Files changed:
  - internal/corpus/runner.go (Transpiler + TranspilerFunc + RunTranspile)
  - internal/corpus/runner_test.go (whole-corpus transpile test)
- Commit: 041152b
- Learnings for future iterations:
  - corpus tests run with cwd = internal/corpus; reuse the existing repoRoot
    const "../.." (declared in generate_test.go, same package) — do not redeclare

  - No import cycle: internal/pipeline does not import internal/corpus, so corpus

## [2026-06-27T04:10:00Z] - US-004
- Implemented the interface-based check runner (Phase 0.4). Added a Checker
  interface{ Check(src string)([]check.Diagnostic,error) } plus a CheckerFunc
  adapter so the free check.Analyze satisfies it without changing the check
  package. RunCheck(root, Case, Checker) reads each KindCheck case's .goal input,
  runs the checker, and matches diagnostics against inline `// want "substr"`
  markers using the same semantics as internal/check/check_test.go: each marker
  is satisfied by a diagnostic on the SAME line whose Message contains substr;
  any unclaimed Error-severity diagnostic fails the case; Warnings may go
  unclaimed.
- TestCheckRunner drives all 50 check cases from corpus/manifest.json through
  CheckerFunc(check.Analyze); all pass. It t.Fatalf's if the manifest yields zero
  check cases.
- Files changed:
  - internal/corpus/check_runner.go (Checker + CheckerFunc + RunCheck + marker parse)
  - internal/corpus/check_runner_test.go (whole-corpus check test)
  - history/20260627-feature-us-004-interface-based-check-runner/ (initiative)
- Learnings for future iterations:
  - No import cycle: internal/check imports only internal/analyze, so corpus may
    import check (same direction as corpus->pipeline in US-003).
  - The // want marker regexp `//\s*want\s+"([^"]*)"` and OffsetToPosition line
    mapping are reused verbatim from the existing check harness — keep them in
    sync if the marker syntax ever changes.
  - check.Error is the Error severity const; check.Diagnostic carries Pos (byte
    offset), Severity, Code, Message. Use check.OffsetToPosition(src, d.Pos).Line.
  - US-006 will rewire internal/check/check_test.go to delegate to this runner so
    its hardcoded testdata/check path walk disappears.
