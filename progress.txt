# Progress Log — goal AST front-end rewrite

## Codebase Patterns
- Zero-dependency: stdlib only. Tests use stdlib `testing` (NO testify), even
  though the injected go.md rule mentions testify — the project constraint wins.
- New packages live under `internal/`; fixtures go in a sibling `testdata/` dir
  (see internal/{analyze,check,pipeline,corpus}/testdata).
- Verify gates (prd.json verifyCommands): `go build ./...`, `go vet ./...`,
  `go test ./... -count=1`. All must be green before committing.
- Follow REWRITE-ARCHITECTURE.md phasing; §10 details Phases 0–2 (the approved
  scope). Phase 0 = corpus hardening (US-001..US-010).
- Each story runs through the morse-code `feature` workflow in --automode on the
  current branch (no new branch); human-review gates are passed via
  `morse-code workflow resume`.
- The golden corpus is indexed by `corpus/manifest.json` (repo root). Regenerate
  after adding/removing golden files: `go generate ./internal/corpus` (or
  `go run ./cmd/corpus-gen -root .`). Paths are repo-root-relative, slash-form.
- `internal/corpus` imports `internal/pipeline` and `internal/check`. Any test
  that delegates to a corpus runner must therefore live in an EXTERNAL `_test`
  package (`package pipeline_test` / `check_test`) to avoid an import cycle; an
  internal-package test importing corpus will not compile. Internal and external
  test packages coexist fine in one directory. From `internal/<pkg>`, repo root is
  `../..` and the manifest is `../../corpus/manifest.json` (same depth as corpus).
- Corpus cases share Kind across modes: package-mode cases are also
  Kind=transpile. Any loop selecting transpile cases for a FILE-mode runner
  (one that reads Case.Input as a single file) MUST also guard
  `c.Mode == ModeFile`, or it will misread a package fixture's directory Input.
  Package cases run through corpus.RunPackage (Mode=package) only.

## [2026-06-27T03:26:00Z] - US-012
- Added internal/lexer (Phase 1.2), the second stone of the AST front-end: a
  hand-written scanner that turns goal source into []token.Token for the core Go
  subset — identifiers, reserved keywords, int/float/imag/char/string literals,
  the full operator+delimiter set, and // and /* */ comments — each carrying an
  accurate token.Pos{Offset, Line, Col}. Modeled on go/scanner but smaller and
  emitting our token.Token directly.
- API: New(src)*Lexer + (*Lexer).Next() token.Token (incremental, for the future
  parser) and Tokens(src)[]token.Token (whole-slice, for tests). Cursor tracks
  offset/rdOffset/ch plus line + lineStart; Col = offset-lineStart+1, Line bumps
  after each '\n'. Multi-char operators are longest-matched via nested switches;
  unrecognized bytes become ILLEGAL tokens (carrying the byte + Pos) without
  panicking and still advance, guaranteeing progress; the stream ends in EOF.
- SCOPE: goal-specific lexemes are deliberately deferred to US-013 — '?' lexes as
  ILLEGAL, '=>' as ASSIGN then GTR, '.' always as PERIOD (no ELLIPSIS), and '///'
  scans as an ordinary '//' line COMMENT. Do NOT emit QUESTION/FAT_ARROW/ELLIPSIS/
  DOC_COMMENT or attach trivia here; that is US-013's job.
- Files changed:
  - internal/lexer/lexer.go (Lexer + New/Next/Tokens + scan methods)
  - internal/lexer/lexer_test.go (multi-line position test = the AC; plus
    keyword/ident, literals, longest-match operators, ILLEGAL, comments)
  - history/20260627-feature-us-012-lex-core-go-tokens-with-positions/ (initiative)
- Commit: 799f761
- Learnings for future iterations:
  - The multi-line acceptance test pins exact Offset/Line/Col per token by hand;
    when editing the sample, recompute byte offsets carefully (blank lines and
    leading tabs each advance offset by 1; tab is one byte column).
  - next() applies the pending newline's line/col bump at the START of the call
    (when the just-consumed ch was '\n'), so a token's Pos is captured before any
    newline is crossed — keep that ordering if you touch the cursor.
  - US-013 extends THIS lexer: fold '=>'→FAT_ARROW and '...'→ELLIPSIS into
    scanOperator, emit '?'→QUESTION (currently ILLEGAL), and special-case '///'
