# Implementation Plan — tree-sitter-goal (M1)

## Layout (`editors/tree-sitter-goal/`)

```
grammar.js                 # the grammar
src/scanner.c              # minimal external scanner: automatic line terminator (Go ASI)
src/parser.c, src/*        # generated by `tree-sitter generate` (checked in)
queries/highlights.scm     # highlight queries (standard captures)
test/corpus/*.txt          # tree-sitter corpus tests for goal constructs
package.json               # tree-sitter metadata + generate/test scripts
README.md, .gitignore
```

## Build order

1. **grammar.js skeleton** — `source_file`, package/import, then declarations. Get
   `tree-sitter generate` to succeed on a trivial program.
2. **External scanner** (`src/scanner.c`) — emit an automatic terminator at newline/`}`/EOF
   when the previous token can end a statement (Go ASI). Grammar references it via
   `externals: [$._automatic_semicolon]` and ends statements with it. This is the one piece
   that genuinely needs C; raw/rune/interpreted strings are expressible as DSL `token`s.
3. **Declarations** — `func` (with optional receiver + generics), `type` (struct with optional
   `implements`, interface, alias), `enum` (variants, payload `{ field: Type }`), `const`/`var`,
   `from func`, `derive func`, `sealed interface`.
4. **Types** — named, qualified (`pkg.T`), pointer, slice/array, map, generic `Name[T, ...]`;
   `Result[T, E]` and `Option[T]` as named generic types (highlighted specially via queries).
5. **Statements** — block, `return`, `if`/`else`, `for` (3 forms + range), `go`, `defer`,
   short-var `:=`, assignment `=`/op-assign, expression statement, `assert`, `match` statement,
   `name := expr?` (the `?` unwrap binding).
6. **Expressions** — identifier, selector `a.b`, call, index `a[...]`, composite literal
   `T{...}` with keyed elements (`field: expr`) and spreads (`...defaults`/`...derive(e)`),
   unary/binary ops, postfix `?`, parenthesized, literals. `Enum.Variant`, `Result.Ok/Err`,
   `Option.Some/None` fall out of selector/call (highlighted via queries).
7. **`match`** — `match expr { (pattern => body)* }`; pattern is an expression or `_`; arrow `=>`.
8. **Comments** — `//` line, `/* */` block, and `///` doc comment as its own node containing
   `>>>` doctest markers (so queries can color them distinctly).
9. **highlights.scm** — map nodes/keywords to `@keyword`, `@type`, `@type.builtin`,
   `@constant.builtin`, `@function`, `@variable.member` (enum variants), `@operator`,
   `@comment` / `@comment.documentation`, `@string`, `@number`.
10. **Corpus tests** — one `test/corpus/*.txt` per goal construct (enum, match, result/option,
    `?`, spreads, doctest, implements, derive/from).

## Verification (acceptance gates)

- `tree-sitter generate` — no unresolved conflicts (iterate on `prec`/`conflicts`).
- A script parses all 103 `.goal` files; **zero ERROR/MISSING** (the FR-001 bar).
- `tree-sitter test` — corpus tests pass.
- `tree-sitter highlight editors/vscode/examples/sample.goal` — goal tokens get captures.
- Add a `task` target or README commands for the loop.

## Risk handling

- **Conflicts**: composite-literal `T{}` vs block/control braces, and `match`/`switch`. Resolve
  with precedence + the external terminator. If a corpus file ERRORs, add/adjust the rule.
- **ASI correctness**: model the scanner on the well-known Go approach (valid-end-token set).
- Keep `extras` = whitespace + comments; newlines handled by the scanner, not `extras`.

## Traceability

FR-001 → corpus parse gate; FR-002 → highlights.scm; FR-003 → dedicated nodes (enum_declaration,
match_expression/arm, result_type/option_type, unwrap `?`, spread, doc_comment); FR-004 →
scanner.c terminator; FR-005 → generate + checked-in `src/`.
